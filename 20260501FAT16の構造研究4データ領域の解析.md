# 検証用FAT16ディスク及びファイルの生成

```bash
dd if=/dev/zero of=fat16.img bs=1M count=10
sudo mkfs.vfat -F 16 fat16.img
sudo mount -o loop,uid=$(id -u),gid=$(id -g) fat16.img /mnt

echo "TEST FILE1" | sudo tee /mnt/test1.txt
```

# ルートディレクトリ構造体

```c
typedef struct __attribute__((packed)) {
    uint8_t  filename[8];      // ファイル名 (8バイト)
    uint8_t  extension[3];     // 拡張子 (3バイト)
    uint8_t  attributes;       // ファイル属性
    uint8_t  reserved;         // 予約領域 (通常0)
    uint8_t  creation_time_ms; // 作成時刻 (10ms単位)
    uint16_t creation_time;    // 作成時刻 (時/分/秒)
    uint16_t creation_date;    // 作成日付 (年/月/日)
    uint16_t last_access_date; // 最終アクセス日付
    uint16_t first_cluster_hi; // 開始クラスタ上位16bit (FAT16では通常0)
    uint16_t last_write_time;  // 最終更新時刻
    uint16_t last_write_date;  // 最終更新日付
    uint16_t first_cluster_lo; // 開始クラスタ下位16bit
    uint32_t file_size;        // ファイルサイズ (バイト単位)
} fat16_dir_entry_t;
```

```
00005840: 0000 0000 0000 0000 0000 0000 0000 0000  ................

```

開始クラスタ上位16bit = `0000`(0x00)
開始クラスタ上位16bit = `0300`(0x03)

よって`test1.txt`は3クラスタ目から始まる。

```
ブートセクタ＋予約領域: 4セクタ × 512 = 2,048バイト
FAT領域: 2 × 20 × 512 = 20,480バイト
ルートディレクトリ: 32 × 512 = 16,384バイト
合計: 2,048 + 20,480 + 16,384 = 38,912バイト (0x9800) ← クラスタ2開始
```

クラスタ大きさ: 4 × 512 = 2,048バイト
クラスタ3開始: 0x9800 + 2,048 = 0xA000 (40,960バイト) 

データ領域の`test1.txt`が保存されている場所を確認すると、txtファイルの内容が確認できる。

# FAT表とは

ルートディレクトリの`first_cluster_lo`は、データが記録されている初めの1クラスタ(`BPB_SecPerClus`によると今回は4セクタ、2,048バイト)を指している。
よってファイルが1クラスタに収まる場合はfirst_cluster_lo`を読むだけでデータの在り処が分かるが、2セクタ以上になる場合はFAT表を参照することになる。


"HELLO"が連続して5セクタ分(1クラスタを超える)書き込まれたtxtファイルを生成し、FAT表を観察
```
printf 'HELLO%.0s' {1..512} > test_large.txt
sudo cp test_large.txt /mnt/
```

まずはルートディレクトリを確認

```
test@test-fujitsu:~/kaihatsu$ xxd -s 22528 -l 512 fat16.img

```

`test_large.txt`の`first_cluster_lo`の項目を見てみると`0400`(0x04)、よって4セクタ目から始まることが分かる。
次にFAT表の4セクタ目を確認する。 

```
ブートセクタ＋予約領域: 4セクタ × 512 = 2,048バイト (0x0800)
FAT1開始: 0x0800
FAT2開始: FAT1 + FATサイズ = 0x0800 + (20×512) = 0x0800 + 0x2800 = 0x3000
```

```
test@test-fujitsu:~/kaihatsu$ xxd -s 0x0800 -l 512 fat16.img
00000800: f8ff ffff 0000 ffff 0500 ffff 0000 0000  ................
00000810: 0000 0000 0000 0000 0000 0000 0000 0000  ................
00000820: 0000 0000 0000 0000 0000 0000 0000 0000  ................
...(省略)...
```

`text1.txt`は次のクラスタが無いので終端を示す`0xFFFF`だけが記録されている。
`test_large.txt`は2クラスタに跨っている為、次のクラスタが記録されている。

|クラスタ| オフセット | 生データ | 実際の値 | エントリ | 意味 |
|-------|------------|----------|----------|----------|------|
|0| 0x0800 | f8ff | 0xFFF8 | FAT[0] | 記憶媒体種別（固定ディスク） |
|1| 0x0802 | ffff | 0xFFFF | FAT[1] | 予約領域 |
|2| 0x0804 | 0000 | 0x0000 | FAT[2] | 空きクラスタ |
|3| 0x0806 | ffff | 0xFFFF | FAT[3] | test1.txtの終端 |
|4| 0x0808 | 0500 | 0x0005 | FAT[4] | 次はクラスタ5 |
|5| 0x080A | ffff | 0xFFFF | FAT[5] | test_large.txtの終端 |


`test_large.txt`は`0x0004`→`0x0005`→`0xFFFF`の様に繋がっている。
初めの1クラスタ分だけはルートディレクトリに保存されている(first_cluster_lo)。

ファイルがどれだけ大きくなっても、データが連続していなくても`first_cluster_lo`→次のクラスタ→...→`0xFFFF`と追っていけば復元が可能。

# データ領域を確認

4クラスタ目と5クラスタ目を確認すると`test_large.txt`の中身が記録されているのが確認出来る。
